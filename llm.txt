Project: Ondo Real Estate web UI

Purpose:
- Marketing site and dashboard UI for Ondo Real Estate, built as a SPA.

Tech stack:
- React 18 with TypeScript
- Vite for bundling and dev server
- React Router (`src/pages` holds route-level components)
- Tailwind CSS + `tailwindcss-animate`
- Radix UI primitives and shadcn-style components (various `@radix-ui/*`, `cmdk`, `sonner`, `lucide-react`)
- Form handling with `react-hook-form` + `zod`
- Charts with `recharts`

Key npm scripts (see package.json):
- `npm run dev` ‚Äì start Vite dev server
- `npm run build` ‚Äì typecheck and build for production
- `npm run preview` ‚Äì preview built app
- `npm run lint` ‚Äì run ESLint on TypeScript/React code

Source layout (high level):
- `src/pages` ‚Äì top-level routed pages (marketing, auth, dashboard shells, etc.)
- `src/components` ‚Äì shared UI components and building blocks
- `src/features` ‚Äì feature-oriented modules (logic + UI)
- `src/hooks` ‚Äì reusable React hooks
- `src/lib` ‚Äì app-level utilities (API client, auth/session helpers, site metadata)
- `src/constants` ‚Äì static configuration and constant values
- `src/utils` ‚Äì generic helpers

Styling:
- Use Tailwind utility classes for layout, spacing, and typography.
- Prefer existing design tokens/components over hand-rolled inline styles.

Routing:
- Use React Router v6 patterns.
- New top-level routes should be implemented as components in `src/pages` and integrated into the router configuration (typically in the main app entry).

Forms and validation:
- Prefer `react-hook-form` with `zod` schemas for new forms.
- Keep form schemas close to the form component or in the relevant `src/features/*` module.

State and data flow:
- Keep business logic in `src/features` or `src/lib` rather than inside page components when possible.
- Prefer hooks in `src/hooks` for reusable behaviors.

Environment/config:
- Environment variables and required config should be documented in `.env.example`.
- Do not hardcode secrets or environment-specific URLs; use Vite env variables instead (e.g. `import.meta.env.VITE_*`).

LLM-specific guidance:
- Respect existing patterns; before adding new libraries, look for similar usage under `src/features`, `src/components`, or `src/lib`.
- When adding components, keep them small and composable; co-locate tests and stories (if any) next to the component.
- Follow TypeScript strictly (no `any` unless unavoidable and justified with comments).
- Run `npm run lint` and fix new lint errors introduced by changes.

Non-goals:
- Do not modify build tooling (`vite.config.ts`, `tsconfig*.json`, ESLint config) unless explicitly requested.
- Avoid introducing breaking API shape changes to shared helpers in `src/lib` without clear justification.
